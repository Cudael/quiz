## CRITICAL FINDINGS

### 1. **CRITICAL: Weak Cryptographic Randomness in Duel Code Generation**

**File:** [src/server/duel.ts](src/server/duel.ts#L3-L10)  
**Severity:** CRITICAL  
**Type:** Anti-Cheat/Cryptographic Weakness

**Issue:**
```typescript
export function generateDuelCode(randomFn: () => number = Math.random) {
  let result = ''
  for (let i = 0; i < 6; i++) {
    const index = Math.floor(randomFn() * DUEL_CODE_CHARSET.length)
    result += DUEL_CODE_CHARSET[index]
  }
  return result
}
```

Duel codes (6-character identifiers) are generated using `Math.random()`, which is **not cryptographically secure**. An attacker can:
- Predict duel codes if they know the approximate timestamp of code generation
- Systematically enumerate active duel codes to join/interfere with other players' duels
- Perform statistical analysis to reduce the search space from $36^6$ (2.1 billion) possibilities

**Impact:** Duel session hijacking, unauthorized participation, competitive advantage manipulation.

**Proof of Concept:** An attacker can use browser developer tools to seed their own `Math.random()` and predict future code sequences, or implement cryptanalysis of generated codes to guess valid ones within the active duel window.

**Remediation:**
Replace `Math.random()` with `crypto.getRandomValues()`:
```typescript
export function generateDuelCode() {
  const bytes = crypto.getRandomValues(new Uint8Array(6))
  let result = ''
  for (let i = 0; i < 6; i++) {
    result += DUEL_CODE_CHARSET[bytes[i] % DUEL_CODE_CHARSET.length]
  }
  return result
}
```

**Related Code:**  
[src/app/api/duel/create/route.ts](src/app/api/duel/create/route.ts#L42) — Calls `generateDuelCode()` without passing a secure random function.

## MEDIUM FINDINGS

### 1. **MEDIUM: In-Memory Rate Limit Fallback Degrades on Serverless Platforms**

**Files:** [src/server/rate-limit.ts](src/server/rate-limit.ts#L34-L80), [src/server/play-token.ts](src/server/play-token.ts#L110-L150)  
**Severity:** MEDIUM  
**Type:** Rate Limiting / Denial of Service

**Issue:**
When Redis is unavailable (misconfiguration or network failure), rate limiting and nonce tracking fall back to in-memory `Map` storage:

```typescript
// rate-limit.ts
const store = new Map<string, RateLimitEntry>()

function getRedis(): Redis | null {
  if (redis !== null) return redis
  const url = process.env.UPSTASH_REDIS_REST_URL
  const token = process.env.UPSTASH_REDIS_REST_TOKEN
  if (url && token) {
    redis = new Redis({ url, token })
  }
  return redis  // null if env vars missing
}
```

On serverless platforms (Vercel, AWS Lambda), **each function invocation is a separate isolate** with independent memory. A user could:
1. Issue 30 `/api/play/submit` requests across 5 different function instances
2. Each instance's in-memory map would only see 6 requests, allowing 4 requests per instance = **20 total bypasses** of the 30-request-per-5-min limit

**Impact:**
- Score inflation via brute-force answer submission
- Leaderboard manipulation
- Resource exhaustion (excessive database writes)
- Play token replay attacks (nonce tracking also affected)

**Example Attack:**
```
Thread 1 (Instance A): POST /api/play/submit (accepted) → Rate limit: 1/30
Thread 2 (Instance B): POST /api/play/submit (accepted) → Rate limit: 1/30 (different instance!)
... repeat with multiple concurrent requests across instances
Result: 150+ requests processed when limit is 30
```

**Severity Justification:**
- Easy to exploit with concurrent requests
- Enables both anti-cheat bypass and nonce replay attacks
- Platform degradation under load increases likelihood of Redis unavailability

**Remediation:**
1. **Mandatory Redis in production** — Update deployment documentation and `.env.example` to clearly mark `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` as **REQUIRED for production**.
2. **Fail-secure** — Add explicit check and hard-fail if Redis is not available in production:
   ```typescript
   function getRedis(): Redis {
     const url = process.env.UPSTASH_REDIS_REST_URL
     const token = process.env.UPSTASH_REDIS_REST_TOKEN
     if (!url || !token) {
       if (process.env.NODE_ENV === 'production') {
         throw new Error('UPSTASH_REDIS_REST_URL and UPSTASH_REDIS_REST_TOKEN must be set in production')
       }
       // In-memory fallback OK for dev
       return null as any
     }
     return new Redis({ url, token })
   }
   ```
3. **Monitoring** — Add alerts if Redis connection fails in production.

### 2. **MEDIUM: Insufficient Validation of Question Answers in Duel Submit**

**File:** [src/app/api/duel/[id]/submit/route.ts](src/app/api/duel/[id]/submit/route.ts#L77-L104)  
**Severity:** MEDIUM  
**Type:** Input Validation / Cheating

**Issue:**
The duel submit endpoint re-queries **all published questions** to validate answers, but does not validate that submitted question IDs actually belong to the duel's originally selected question set:

```typescript
const candidateQuestions = await prisma.question.findMany({
  where: {
    quiz: {
      isPublished: true,
      ...(duel.categoryId ? { categoryId: duel.categoryId } : {}),
    },
  },
  // ... fetch all questions matching duel category
})

const selectedQuestionIds = pickDuelQuestionIds(
  candidateQuestions.map((question) => question.id),
  duel.id,
  duel.questionCount
)
```

**Attack Scenario:**
1. Duel A uses category "Science" and generates questions via deterministic shuffle (`pickDuelQuestionIds`)
2. Player B submits answers for **different questions** (e.g., easier questions from the same category)
3. The endpoint validates those alternate questions because:
   - They exist in the category
   - They're part of the `candidateQuestions` set
   - The score is computed based on submitted questions, not the duel's original set

**Result:** Player B scores on questions they shouldn't have been asked, skewing leaderboard.

**Current Code Flow:**
```typescript
// Does NOT verify the submitted questionIds match the duel's original set
for (const question of selectedQuestions) {
  const submitted = answersByQuestion.get(question.id)  // accepts ANY matching question
  if (!submitted) continue
  // ... score calculation
}
```

**Remediation:**
Store the selected question IDs in the `Duel` record when the duel starts (or pass them in the `/start` response), then validate submitted answers against that immutable set:

```typescript
// In /api/duel/[id]/start/route.ts:
const questionIds = pickDuelQuestionIds(
  candidateQuestions.map(q => q.id),
  duel.id,
  duel.questionCount
)
await prisma.duel.update({
  where: { id: duel.id },
  data: {
    status: 'IN_PROGRESS',
    selectedQuestionIds: questionIds // NEW field
  }
})

// In submit route:
const duel = await prisma.duel.findUnique({
  where: { id },
  select: { 
    id: true, 
    selectedQuestionIds: true,  // fetch stored IDs
    // ...
  }
})

// Validate submitted questions match the duel's original set
const allowedQuestionIds = new Set(duel.selectedQuestionIds)
for (const answer of parsed.data.answers) {
  if (!allowedQuestionIds.has(answer.questionId)) {
    return NextResponse.json({ error: 'Invalid question for this duel' }, { status: 400 })
  }
}
```

### 3. **MEDIUM: Race Condition in Duel Participant Join**

**File:** [src/app/api/duel/join/route.ts](src/app/api/duel/join/route.ts#L25-L48)  
**Severity:** MEDIUM  
**Type:** Race Condition / State Consistency

**Issue:**
The join endpoint checks if a participant already exists, but does not prevent the same user from joining multiple times in a race condition:

```typescript
const existing = await prisma.duelParticipant.findFirst({
  where: {
    duelId: duel.id,
    ...(session?.user?.id ? { userId: session.user.id } : { guestKey }),
  },
  select: { id: true },
})
if (existing) {
  return NextResponse.json({ error: 'Already joined' }, { status: 400 })
}

// RACE CONDITION: Between the check above and the create below,
// another request for the same user could also pass the check

await prisma.duelParticipant.create({
  data: {
    duelId: duel.id,
    userId: session?.user?.id ?? null,
    guestName: session?.user?.id ? null : 'Guest',
    guestKey: session?.user?.id ? null : guestKey,
  },
})
```

**Attack Scenario:**
1. User A sends 2 rapid requests to `/api/duel/join` with code "ABC123"
2. Request 1: `findFirst` returns null (not yet joined) → proceeds to create
3. Request 2: `findFirst` also returns null (create hasn't committed yet) → also proceeds to create
4. Both requests try to create `DuelParticipant` for the same user
5. Result: User A participates twice (inflated score count, double XP if scoring isn't guarded)

**Remediation:**
Use a **unique constraint** in Prisma schema and handle the conflict:

```prisma
// schema.prisma
model DuelParticipant {
  // ...
  @@unique([duelId, userId])
  @@unique([duelId, guestKey])
}
```

Then handle the unique constraint violation:
```typescript
try {
  await prisma.duelParticipant.create({
    data: {
      duelId: duel.id,
      userId: session?.user?.id ?? null,
      guestKey: session?.user?.id ? null : guestKey,
    },
  })
} catch (error) {
  if (error instanceof Prisma.PrismaClientKnownRequestError && error.code === 'P2002') {
    return NextResponse.json({ error: 'Already joined' }, { status: 400 })
  }
  throw error
}
```

## ANTI-CHEAT & SCORING INTEGRITY — DETAILED ANALYSIS

### ✅ **Play Token System: SECURE**

**Implementation:** [src/server/play-token.ts](src/server/play-token.ts)

**Strengths:**
- ✅ HMAC-SHA256 signature verification prevents tampering
- ✅ Per-request nonce prevents replay attacks (consumed on first use)
- ✅ 4-hour TTL bounds token validity window
- ✅ Nonce tracked via Redis with in-memory fallback (see MEDIUM #1 for fallback concern)
- ✅ Token contains quizId—cannot swap tokens between quizzes

**Code Evidence:**
```typescript
// src/server/play-token.ts:180-195
export async function verifyPlayToken(
  token: string,
  expectedQuizId: string
): Promise<{ valid: boolean; quizId?: string }> {
  try {
    const [payloadB64, sigB64] = token.split('.')
    if (!payloadB64 || !sigB64) return { valid: false }

    const key = await importKey(getSecret())
    const ok = await verify(payloadB64, sigB64, key)  // HMAC verification
    if (!ok) return { valid: false }

    const payload: PlayTokenPayload = JSON.parse(
      Buffer.from(payloadB64, 'base64url').toString('utf8')
    )
    if (Date.now() - payload.issuedAt > TOKEN_TTL_MS) return { valid: false }
    if (payload.quizId !== expectedQuizId) return { valid: false }
    
    if (!(await consumeNonce(payload.nonce, expiresAt))) return { valid: false }  // Replay check
    return { valid: true, quizId: payload.quizId }
  } catch {
    return { valid: false }
  }
}
```
