## FINDINGS BY SEVERITY

### 🔴 CRITICAL

#### **1. Timing Attack on CRON_SECRET Comparison**

**File:** [src/app/api/cron/cleanup-guests/route.ts](src/app/api/cron/cleanup-guests/route.ts#L10-L16)  
**Lines:** 10–16

**Issue:**
```typescript
function isAuthorizedCronRequest(request: Request): boolean {
  const secret = process.env.CRON_SECRET
  if (!secret) {
    return false
  }

  const authorization = request.headers.get('authorization')
  return authorization === `Bearer ${secret}`  // ← VULNERABLE
}
```

The string comparison `authorization === 'Bearer ${secret}'` is **not timing-safe**. An attacker can measure response time differences to determine correct characters of the secret, enabling character-by-character brute-force attacks. This is a practical vulnerability because:

- The comparison length varies based on the secret  
- An attacker can send multiple requests with manipulated `Authorization` headers
- Even millisecond-level timing differences are measurable over the network

**Evidence:** No use of `crypto.timingSafeEqual()` anywhere in the codebase (grep confirms).

**Remediation:** Use `crypto.timingSafeEqual()`:
```typescript
import { timingSafeEqual } from 'crypto'

function isAuthorizedCronRequest(request: Request): boolean {
  const secret = process.env.CRON_SECRET
  if (!secret) {
    return false
  }

  const authorization = request.headers.get('authorization') || ''
  const expected = `Bearer ${secret}`
  
  // Both buffers must be the same length for timingSafeEqual
  if (authorization.length !== expected.length) {
    return false
  }
  
  try {
    return timingSafeEqual(Buffer.from(authorization), Buffer.from(expected))
  } catch {
    return false
  }
}
```

**Impact:** An attacker can gradually reveal the CRON_SECRET and invoke the cleanup endpoint to delete guest data, potentially disrupting service or data loss.

### 🟠 MEDIUM

#### **1. Email Enumeration via HTTP Status Codes in Register Endpoint**

**File:** [src/app/api/auth/register/route.ts](src/app/api/auth/register/route.ts#L40-L84)  
**Lines:** 40–84

**Issue:**
```typescript
const existingUser = await prisma.user.findUnique({
  where: { email },
  select: { id: true },
})

if (existingUser) {
  return NextResponse.json({ error: 'Unable to register account.' }, { status: 409 })
  // ↑ Returns 409 Conflict
}

try {
  const passwordHash = await hashPassword(password)
  await prisma.user.create({
    data: { /* ... */ },
    select: { id: true },
  })
  // ...
} catch (error) {
  if (error instanceof Prisma.PrismaClientKnownRequestError && error.code === 'P2002') {
    return NextResponse.json({ error: 'Unable to register account.' }, { status: 409 })
    // ↑ Also 409 on constraint violation
  }
  return NextResponse.json({ error: 'Unable to register account.' }, { status: 400 })
  // ↑ Returns 400 for other errors
}
```

While the error **message** is identical (`"Unable to register account."`), the **HTTP status code** differs:
- `409 Conflict` → User already exists  
- `400 Bad Request` → Validation/other errors

An attacker can enumerate valid email addresses by checking the status code:
```javascript
// Attacker's reconnaissance
const response = await fetch('/api/auth/register', {
  method: 'POST',
  body: JSON.stringify({ 
    name: 'Test', 
    email: 'existing@example.com',  // To enumerate
    password: 'StrongPass123!'
  })
})

if (response.status === 409) {
  console.log('Email exists!')
} else if (response.status === 400) {
  console.log('Email is new')
}
```

**Contrast:** The [forgot-password endpoint](src/app/api/auth/forgot-password/route.ts#L21-L28) correctly returns `200 Ok` regardless of whether the email exists—preventing enumeration.

**Remediation:** Return `400` for all validation/registration failures:
```typescript
if (existingUser) {
  return NextResponse.json({ error: 'Unable to register account.' }, { status: 400 })
}

// And in the catch block:
if (error instanceof Prisma.PrismaClientKnownRequestError && error.code === 'P2002') {
  return NextResponse.json({ error: 'Unable to register account.' }, { status: 400 })
}
```

**Impact:** Low. Attackers can enumerate a list of registered email addresses over time, useful for phishing campaigns or account targeting.

#### **2. Race Condition in Password Reset Token Consumption**

**File:** [src/app/api/auth/reset-password/route.ts](src/app/api/auth/reset-password/route.ts#L28-L67)  
**Lines:** 28–67

**Issue:**
```typescript
const verificationToken = await prisma.verificationToken.findUnique({
  where: { token: tokenHash },
  select: { identifier: true, expires: true, token: true },
})

if (!verificationToken || !verificationToken.identifier.startsWith('reset:')) {
  return NextResponse.json({ error: 'Invalid or expired reset link.' }, { status: 400 })
}

if (verificationToken.expires < new Date()) {
  await prisma.verificationToken.delete({
    where: {
      identifier_token: {
        identifier: verificationToken.identifier,
        token: verificationToken.token,
      },
    },
  })
  return NextResponse.json(
    { error: 'Reset link has expired. Please request a new one.' },
    { status: 400 }
  )
}

// Extract email and update password
const email = verificationToken.identifier.slice('reset:'.length)
const passwordHash = await hashPassword(newPassword)
await prisma.user.update({
  where: { email },
  data: { passwordHash },
})

// Delete token AFTER password update ← Race condition window
await prisma.verificationToken.delete({
  where: {
    identifier_token: {
      identifier: verificationToken.identifier,
      token: verificationToken.token,
    },
  },
})
```

Between the password update (line ~55) and token deletion (line ~62), a **race condition window** exists:

1. Attacker finds token `reset:user@example.com`  
2. Request A: Gets token, updates password successfully ← Returns before deletion  
3. Request B (concurrent): Gets same token, updates password again to attacker-controlled value  
4. Request A now deletes the token  
5. Request B completes with attacker's password

This requires precise timing but is feasible on high-latency networks or with connection pooling delays.

**Remediation:** Delete token atomically within the same transaction:
```typescript
await prisma.$transaction(async (tx) => {
  await tx.user.update({
    where: { email },
    data: { passwordHash },
  })

  await tx.verificationToken.delete({
    where: {
      identifier_token: {
        identifier: verificationToken.identifier,
        token: verificationToken.token,
      },
    },
  })
})
```

**Evidence:** [src/server/auth.ts](src/server/auth.ts#L56-L106) uses transactions for OAuth account linking, confirming the pattern is known and used elsewhere.

**Impact:** Low. An attacker would need a valid reset token (typically via email interception/account compromise) and precise timing to hijack a password reset.

#### **1. In-Memory Rate Limiter Bypass on Serverless (Documented Limitation)**

**File:** [src/server/rate-limit.ts](src/server/rate-limit.ts#L44-L77)  
**Lines:** 44–77, **Comment:** Line 50–56

**Issue:**
```typescript
/**
 * Returns `true` when the request should be allowed, `false` when it exceeds
 * the limit.
 *
 * Uses Upstash Redis for shared counters across serverless instances when the
 * `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` env vars are set;
 * falls back to an in-memory fixed-window counter otherwise.
 */
export async function checkRateLimit(key: string, config: RateLimitConfig): Promise<boolean> {
  const client = getRedis()

  if (client) {
    // Redis handles cross-instance rate limiting
    try {
      const redisKey = `rl:${key}`
      const count = await client.incr(redisKey)
      if (count === 1) {
        await client.pexpire(redisKey, config.windowMs)
      }
      return count <= config.limit
    } catch {
      // Fall through to in-memory
    }
  }

  // In-memory fixed-window fallback
  const now = Date.now()
  const entry = store.get(key)
  // ...
}
```

When Redis is unavailable or not configured, the in-memory `store` Map is local to each serverless instance. On Vercel:

1. Attacker sends 5 requests to instance A → rate limited  
2. Attacker sends 5 requests to instance B (cold start) → succeeds  
3. Attacker sends 5 requests to instance C (different container) → succeeds

**Status:** **This is documented behavior** with a fallback warning log. The code includes a clear comment explaining the serverless limitation. Not a design flaw, but a known trade-off.

**Evidence:** Documented in the function docstring (lines 50–56).

export function getClientIp(request: Request): string {
  const realIp = request.headers.get('x-real-ip')
  if (realIp) return realIp.trim()

  if (forwarded) {

#### **8. ✅ Redirect Validation (safeCallbackUrl)**

**File:** [src/lib/safe-callback-url.ts](src/lib/safe-callback-url.ts)

**Evidence:**
```typescript
export function safeCallbackUrl(callbackUrl: string | undefined, defaultUrl: string): string {
  if (!callbackUrl) return defaultUrl

  // Allow relative paths but NOT protocol-relative URLs like "//evil.com"
  if (callbackUrl.startsWith('/') && !callbackUrl.startsWith('//')) {
    return callbackUrl
  }

  const base = process.env.NEXTAUTH_URL
  if (base) {
    try {
      const target = new URL(callbackUrl)
      const origin = new URL(base)
      if (target.origin === origin.origin) {
        return callbackUrl
      }
    } catch {
      // Not a valid absolute URL — fall through to default.
    }
  }

  return defaultUrl
}
```

✅ **Rejects protocol-relative URLs** (`//evil.com` is explicitly blocked).  
✅ **Rejects cross-origin URLs** (compares `origin`, not just `host`).  
✅ **Rejects malformed URLs** (URL constructor throws, caught and ignored).  
✅ **Safe fallback** to `defaultUrl` for all invalid cases.
